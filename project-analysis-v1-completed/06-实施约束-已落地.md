# 实施约束 — 已落地

> **V1 已强制执行**的硬约束摘要。

---

## 1. 工程与产品命名

| 项 | 固定值 |
|----|--------|
| 产品名 | `shellder-agent` |
| Web 工程 | `shellder-web-console` |
| 后端 | `shellder-agent-server` |
| Worker | `shellder-job-worker` |

---

## 2. 租户模型（）

- 租户在 **Agent 平台 MySQL 内维护**，不从上层业务系统同步
- 业务表 `tenant_id` → `tenant.id`
- 可选 `externalTenantId` 供 OpenAPI/Copilot 映射
- OpenAPI 可传 `tenantId` 或 `externalTenantId`

---

## 3. 模型接入与 wiki 边界

1. **平台主 LLM** 在 `shellder-agent-server` 配置（`/api/v1/settings/llm`）
2. **wiki 知识库服务** 仅：四层内容、媒体、**`dialogue/recall`（仅召回）**
3. 问答型：**recall（wiki）→ generate（平台 LLM）**
4. **禁止** wiki/pathy 侧 LLM 作为 shellder 生产主推理
5. **OpenAPI 应用接入** 与 **模型接入** 是两套能力，不得混名

配置键：`knowledge.wikiBaseUrl`（见 `docs/knowledge-wiki-integration.md`）。

---

## 4. Prompt 管理

1. 业务代码**禁止** Prompt 正文；仅 `prompt_key` + 变量 + `PromptResolver`
2. 业务表**禁止**存 LLM Prompt 全文；允许 `prompt_binding`
3. Runtime / Copilot / OpenAPI **仅消费 `published`**
4. 新增 key 须登记 `prompt-keys.ts` 与 seed SQL
5. Prompt 模块**不调用 LLM**（试跑接口委托 `LlmService`）

预置 key 见 `src/prompt/prompt-keys.ts`（含 `qa.*`、`query.*`、`connector.er_*`）。

---

## 5. 前端 UI

- 主组件库：**Ant Design**
- Tailwind 用于布局与少量定制
- 现网构建：**Vite + React**（非 Next.js）

---

## 6. 数据库演进

- 按 `project-sql/{模块}/` 递增 SQL
- 新模块须读前序 SQL，禁止无说明重写已交付结构

---

## 7. 查询型执行原则（V1 已落地）

- 仅只读 SQL
- 基于已配置数据源与 ER 发布版
- 限制返回行数、执行时长、表/字段范围（黑名单 + 数据范围）

---

## 8. 改造检查清单

改 LLM 相关逻辑时：

- [ ] 是否经 `PromptResolver` + `published`？
- [ ] 是否经 `LlmService` 而非 wiki LLM？
- [ ] 问答是否两阶段 recall + generate？

改查询型时：

- [ ] 是否校验 ER `published`？
- [ ] SQL 是否经 `SqlToolService`？
- [ ] NL2SQL / 结果解读是否用对应 `prompt_key`？

改租户/权限时：

- [ ] 写操作是否带 `tenant_id` 且校验 RBAC？
- [ ] 写操作是否 `@Audit`？

改 Copilot / OpenAPI 嵌入时：

- [ ] 会话 / 消息 / 审批 / 任务 API 是否校验 **用户或会话归属**（不仅 `tenantId`）？
- [ ] `domainWhitelist` 是否在换票或 Widget 入口 enforcement？

改查询型 / SQL 执行时（补充 §7）：

- [ ] 是否禁止 `UNION` 等子查询结构逃逸？
- [ ] 命名参数绑定是否避免误替换字符串字面量？
- [ ] NL2SQL 语义审核失败是否 **拒绝执行**（fail-closed）？

---

## 9. Copilot / OpenAPI 用户隔离（目标态）

V1 已落地 **租户级** 隔离与独立 JWT issuer；以下为用户级目标约束，现网部分 API 尚未完全满足（见 [01-范围与边界 §6](./01-范围与边界.md#6-已知缺口与后续强化)）：

1. **会话**：`GET/PATCH/DELETE /copilot/v1/sessions/:id` 须 `session.userId === JWT.sub`（已落地）
2. **待确认**：`GET /copilot/v1/confirmations` 仅返回 **当前用户 / 会话** 相关记录；`POST .../confirmations/:id` 须校验审批归属
3. **任务**：`GET /copilot/v1/tasks/:id` 须校验任务关联会话的 `userId`
4. **域名**：`copilot_config.domainWhitelist` 须在换票或 Widget 请求时校验 `Origin` / `Referer`
5. **Token 传递**：SSE 避免长期 JWT 出现在 URL Query（优先短期 ticket 或 Header）

管理端 OpenAPI 应用审批若暴露给终端用户，适用相同归属原则。

---

## 10. SQL 只读执行安全边界（目标态）

在 §7 原则之上，`SqlToolService` 须满足：

1. 子查询包裹前校验括号平衡；**禁止** `UNION` / `INTERSECT` / `EXCEPT` 等多语句结构逃逸
2. `:name` 命名参数替换须 SQL 感知，**不得**替换字符串字面量内的 `:token`
3. 表黑名单检测应与 NL2SQL `referencedTables` 交叉校验（防 `extractTables` 粗解析漏检）
4. 连接器 SQL 测试（管理端）与 Tool 运行时黑名单策略差异须在文档与权限上明确边界

---

## 11. 集中配置

1. **唯一配置根**：`shellder-agent/config/`（npm 包 `@shellder/config`），**禁止**在三端工程内散落硬编码默认值（SQL `maxRows`、能力超时等）
2. **读取方式**：`import { applicationProperties } from '@shellder/config'` → `applicationProperties.get().app.*`
3. **本地开发**：`config/.env` + `config/application.yml`（由 `scripts/setup-local-config.sh` 或 `npm run dev:*` 首次从 `*.example` / `*.dockeruse` 复制）
4. **Docker / 部署**：读取 `config/.env.dockeruse` + `config/application.yml.dockeruse`（`SHELLDER_CONFIG_SOURCE=docker`）
5. **Web 默认值**：`shellder-web-console/src/config/env.ts`（`webEnvConfig`）由 Vite 构建时注入 YAML 片段，与后端 `app.sql` / `app.basic` 对齐
6. **与 system_config 关系**：租户可编辑项（如 `knowledge.wikiBaseUrl`、LLM 接入）仍走 `system_config` 表；YAML 提供 **平台级默认与基础设施** 配置
