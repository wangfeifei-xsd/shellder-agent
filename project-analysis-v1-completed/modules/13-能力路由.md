# 模块：能力路由

| 属性 | 值 |
|------|-----|
| 初始方案 | 功能清单 §1.4 / 架构 Capability Routing |
| 菜单 key | `routing`（RBAC `routing.manage`） |
| 状态 | **已产品化**（两阶段路由 + 规则 AI 辅助 + 条件测试/优化 + 能力自动补齐） |

---

## 信息架构

```
能力路由
├── 路由规则        /routing/rules           → 能力内 routing_rule（keywords/patterns/intents/toolIds/toolKind）
├── 路由测试        /routing/test            → Stage1 + Stage2 联调（含模拟定向）
└── 能力目录        /routing/capabilities    → 四类能力 + dependentTools
```

| 能力 type | Tool 层 | 说明 |
|-----------|---------|------|
| `qa` | 无 Tool | 问答型，不走 Tool 编排 |
| `query` | `ToolType.query` | **仅 NL2SQL**（智能问数） |
| `action` | `action` / `notification` / **`http_query`** | HTTP 写操作 + HTTP 业务只读查询 |
| `workflow` | `workflow` | 流程型 |

**边界（D3）**：`ToolType.http_query` 归属 **action** 能力，**不是** `query` 能力。

---

## 两阶段路由

| 阶段 | 方法 | 职责 |
|------|------|------|
| Stage1 | `routeCapability()` | 跨类型 rules + 启发式；可选 LLM（`routing.classify.system`，默认关） |
| Stage2 | `routeWithinCapability()` | 能力内 `routing_rule` → toolIds；action 内可选 HTTP 查询信号 |
| 编排 | `routeFull()` | Runtime / 路由测试统一入口 |

**Copilot routingMode**（`copilot_config.features`）：

| mode | 创建会话 | 首条消息 |
|------|----------|----------|
| `auto`（默认） | capabilityType 可不传 | Stage1 自动 + Stage2 |
| `pinned` | capabilityType **必填** | 跳过 Stage1 + Stage2 |
| `hybrid` | 可不传 | Tab 可覆盖 Stage1 |

## 核心 API

```
GET/POST/PATCH/DELETE  /api/v1/routing-rules
GET/POST/PATCH/DELETE  /api/v1/capabilities
POST                   /api/v1/routing/test                    # 全链路（typeStage + intraStage）
POST                   /api/v1/routing/test/intra              # 仅 Stage2
POST                   /api/v1/routing-rules/ai-suggest        # AI 从零生成规则草案
POST                   /api/v1/routing-rules/test-conditions   # 即时测试 conditions 匹配
POST                   /api/v1/routing-rules/ai-optimize-conditions  # 未命中时 LLM 优化条件
```

Runtime 内调用 `RoutingEngineService.routeFull()`；结果写入 `routing_result` 系统消息（含 `typeStage` / `intraStage`）。

## 能力目录自动补齐

租户首次使用能力路由时，若 `capability` 表无记录，按 `tenant.config.capabilities` 自动创建四类基础能力（`ensureDefaultCapabilities`）：

| type | 默认名称 |
|------|----------|
| qa | 通用问答 |
| query | 数据查询 |
| action | 业务操作 |
| workflow | 流程编排 |

触发时机：能力列表查询、`getEnabledByTenant()`、路由引擎加载时。

## `resolveToolIds` 优先级

1. `intraStage.toolIds`（能力内规则或信号命中）
2. `candidates[].toolIds`（Stage1 附带，兼容）
3. `capability.dependent_tools`
4. 按 `capabilityType` + `toolKind` fallback 第一个 enabled Tool

action 分支：`toolKind` 为 `http_query` / `action` / `notification` 时分流。

## 路由规则 conditions

```json
{
  "keywords": ["订单"],
  "patterns": ["^查.*订单"],
  "intents": ["order_query"],
  "toolKind": "http_query"
}
```

| 字段 | 匹配规则 | 说明 |
|------|----------|------|
| `keywords` | 子串命中 +10/项 | 用户输入包含即得分 |
| `patterns` | 正则命中 +20/项 | 非法正则会标记 invalidPatterns |
| `intents` | 子串命中 +15/项 | 保留字段，供 NLU 扩展；AI 生成/优化均输出 |
| `toolKind` | — | 仅 **action** 能力可选；限定绑定 Tool 类型 |
| `minScore` | — | 类型已预留，引擎**尚未启用** |

匹配算法见 `routing-conditions.util.ts` → `evaluateRoutingConditions()`。

## 管理端：路由规则 AI 与调试

| 能力 | API | UI（`/routing/rules`） |
|------|-----|------------------------|
| **AI 生成规则** | `ai-suggest` | 描述场景 + 可选示例问法 → 填入 name/keywords/patterns/intents |
| **测试匹配** | `test-conditions` | 编辑 Drawer →「测试匹配」Modal，返回 score / 命中明细 |
| **LLM 优化条件** | `ai-optimize-conditions` | 测试未命中 →「LLM 优化条件」→ 增补 conditions 并回填表单 |

全链路联调仍用 **路由测试** 页（`/routing/test`），与 Runtime 共用 `routeFull`。

## Policy 与确认（2026-06）

| 来源 | 字段 / 行为 |
|------|-------------|
| 路由规则 | `routing_rule.needConfirmation` |
| Policy | Tool 上下文评估 `need_confirm` / 高风险等 |
| 合并 | `RoutingEngineService`：`needConfirmation = matchedRule.needConfirmation \|\| policyNeedConfirm` |

**命中留痕**（与 Policy 菜单 `/rule-hits` 联动）：

- Runtime 在路由级确认**中断前**调用 `persistPolicyHitsForContext`
- 无 Policy 命中时写入 `[路由] {routing_rule.name}`（`rule_id=null`）
- 路由测试 / `checkNeedConfirmation` 预览：`persistHits: false`，不落库

详见 [16-规则与Policy.md](./16-规则与Policy.md)。

## E2E 验收清单

### 路由规则页

- [ ] AI 生成规则草案 → 保存 → 路由测试命中
- [ ] 「测试匹配」对 keywords/patterns 返回与引擎一致的 score
- [ ] 未命中问句 → LLM 优化条件 → 再次测试命中

### 路由测试页

- [ ] 无 `pinnedCapabilityType`：NL2SQL 问句 → Stage1=`query`
- [ ] 模拟定向 + `action`：HTTP 业务问句 → Stage2 命中 `http_query` toolIds
- [ ] `/routing/test` 与 Runtime 对同一输入的 `typeStage` + `intraStage` 一致

### 能力目录

- [ ] 新租户首次打开能力目录 / 路由 → 自动补齐四类基础能力

### Copilot 嵌入

- [ ] **auto**（默认）：首条消息自动路由
- [ ] **pinned**：创建会话须传 `capabilityType`

### 与工具方案联调

- [ ] HTTP 业务查询（规则或信号）→ **action** + `http_query` Tool 执行
- [ ] Policy `needConfirmation` 在路由级与 Tool 级不退化
